## 1. Problem Statement

The vibe-coding-kit currently improves through a single channel: **Stephen notices something is wrong during a session and initiates a kit improvement prompt.**

This creates several failure modes:

- **Friction that never rises to a formal complaint** — Copilot hesitates, looks something up twice, or produces a slightly wrong first attempt. The session continues. The friction is forgotten.
- **Consumer repo siloing** — PIE, DailyInventoryPWA, and other consumer repos each experience kit friction independently. A rule that is confusing in PIE is probably also confusing in DailyInventoryPWA, but there is no mechanism to correlate the signals.
- **Retrospective bias** — Improvements happen after the pain is felt acutely. Mild recurring friction (the kind that compounds over months) may never reach a threshold that prompts action.
- **Single-point observation** — Only Stephen can initiate improvements. Copilot, the agent that encounters the rules most directly, has no structured output channel for surfacing improvement ideas.

**Goal:** Create a structured, low-friction mechanism by which every consumer repo session can contribute friction signal to a shared pool, and that pool can be periodically synthesized into improvement briefs — closing the feedback loop between kit usage and kit improvement.

## 2. Proposed Architecture — The Shared Feedback Directory

### 2.1 Core Concept

Create a **shared friction log directory** at:

```
C:\Users\schur\workspaces\.kit-feedback\
```

This directory sits **outside all git repos** — it is not inside PIE, DailyInventoryPWA, vibe-coding-kit, or any other repo. It has no git history. It is a local aggregation point on Stephen's machine.

Every consumer repo's Copilot session writes friction entries to a repo-specific file inside this directory. Claude (Cowork) analyzes the directory periodically and generates improvement briefs.

```
C:\Users\schur\workspaces\.kit-feedback\
├── README.md                         ← Explains the directory; generated by kit setup
├── PIE-friction-log.md               ← Written by Copilot sessions in PIE
├── DailyInventoryPWA-friction-log.md ← Written by Copilot sessions in DailyInventoryPWA
├── AngularBlackjack-friction-log.md  ← Written by Copilot sessions in AngularBlackjack
├── vibe-coding-kit-friction-log.md   ← Written by Copilot sessions in the kit itself
└── SYNTHESIS-REPORTS\
    ├── SYNTHESIS-2026-04-16.md       ← Generated by periodic analysis
    └── SYNTHESIS-2026-05-01.md       ← Next synthesis run
```

### 2.2 Why Outside Git Repos

| Option | Problem |
|--------|---------|
| Inside each consumer repo | Friction files would not be visible across repos; git subtree pull would risk overwriting |
| Inside vibe-coding-kit repo | Consumers cannot write back to the kit repo (subtree is one-way) |
| Shared network path / cloud | Adds infrastructure dependency; overkill for a single-developer setup |
| **Shared local directory (workspaces/.kit-feedback/)** | **No git complications. Always accessible. All repos can write to it. Claude can read it.** |

The `workspaces/` directory already contains all consumer repos as siblings. A `.kit-feedback/` directory at this level is structurally clean and permanently accessible to both Copilot (via file write) and Claude (via Cowork file access).

## 3. Friction Entry Format

Each friction log file is append-only markdown. Copilot adds a new entry block at the bottom of the file after each session where friction was encountered. Each entry uses a consistent, machine-parseable structure.

### 3.1 Entry Schema

```markdown
---
## FRICTION-ENTRY
- **Date:** YYYY-MM-DD
- **Session PROMPT-ID:** (the PROMPT-ID of the session or prompt where friction occurred; or "UNTRACKED" if session had no formal PROMPT-ID)
- **Kit Version:** (from VIBE-CODING.VERSION.md at time of session)
- **Consumer Repo:** (repo name, e.g. DailyInventoryPWA)
- **Friction Type:** AMBIGUOUS | CONTRADICTORY | MISSING | SLOW | BLOCKING | OTHER
- **Severity:** LOW | MEDIUM | HIGH
- **Location:** (file path and approximate line or section, e.g. protocol/protocol-v7.md § Prompt Review Gate)
- **Description:** (1–4 sentences: what was unclear, contradictory, or slow; what Copilot had to do to work around it)
- **Suggested Fix:** (optional: Copilot's suggestion for how the rule could be clearer or better structured)
- **Resolved This Session:** YES | NO
---
```

### 3.2 Friction Types Defined

| Type | Meaning |
|------|---------|
| **AMBIGUOUS** | The rule exists but its meaning was unclear; required re-reading or inference |
| **CONTRADICTORY** | Two rules or two parts of one rule appear to conflict |
| **MISSING** | A situation arose for which no rule exists; Copilot had to guess |
| **SLOW** | The rule is clear but the required ceremony was disproportionate to the task risk |
| **BLOCKING** | A gate returned BLOCKED for a reason that appeared incorrect or overly strict |
| **OTHER** | Anything that doesn't fit the above; describe in Description field |

### 3.3 Severity Defined

| Severity | Meaning |
|----------|---------|
| **LOW** | Minor friction; session continued normally; may not need fixing |
| **MEDIUM** | Noticeable friction; Copilot had to pause or ask for clarification |
| **HIGH** | Session was meaningfully slowed or a wrong action was nearly taken before the ambiguity was caught |

### 3.4 Example Entry

```markdown
---
## FRICTION-ENTRY
- **Date:** 2026-04-16
- **Session PROMPT-ID:** PWA-EPIC004-S03-IMPLEMENT-001
- **Kit Version:** v7.3.2
- **Consumer Repo:** DailyInventoryPWA
- **Friction Type:** CONTRADICTORY
- **Severity:** HIGH
- **Location:** protocol/protocol-v7.md line 90 — Doc Audit Sequencing paragraph
- **Description:** The paragraph states "occurs AFTER Proof-of-Read" in sentence 1 and "work prompts in a fresh session must run Doc Audit first" in sentence 2. These are contradictory. I initially interpreted Doc Audit as per-prompt (sentence 1) and began prompting without a session-start run. The contradiction was caught during Proof-of-Read review.
- **Suggested Fix:** Explicitly label Doc Audit as a "session-level gate, not a per-prompt gate" and provide a numbered sequence showing session start → first prompt → post-commit as distinct phases.
- **Resolved This Session:** NO
---
```

## 4. What Copilot Must Be Instructed To Do

For this system to work, Copilot needs an explicit instruction in `copilot-instructions-v7.md`. Without this, friction logging will not happen reliably.

### 4.1 Proposed Instruction Addition (copilot-instructions-v7.md)

```markdown
## Friction Log — Mandatory at Session End

Before closing any session (as part of the End-of-Session contract), check if any of the following occurred:
- A rule required re-reading more than once before its meaning was clear
- Two rules appeared to conflict with each other
- A situation arose for which no rule existed and you had to infer the correct behavior
- A gate returned BLOCKED or WARN for a reason that felt disproportionate to the risk
- Any session step took significantly more back-and-forth than it should have

If ANY of the above occurred, append a FRICTION-ENTRY to the friction log file for this repo:

**Path:** `C:\Users\schur\workspaces\.kit-feedback\<REPO-NAME>-friction-log.md`

Use the entry schema defined in:
`C:\Users\schur\workspaces\vibe-coding-kit\docs\status\REPORT-KIT-SELF-LEARNING-FRICTION-RESEARCH-001.md § 3.1`

If no friction occurred, write nothing. Do not create a "FRICTION-ENTRY: none" entry — silence is the signal.
```

### 4.2 Instruction Placement in Copilot Flow

The friction log instruction should appear in the **End-of-Session** section of copilot-instructions, after the Green Gate and Completion Report requirements. It must appear BEFORE the session is considered closed, so it is not skipped when a session ends quickly.

### 4.3 The Risk of Low-Quality Entries

If the instruction is too broad, Copilot may log friction for every minor uncertainty, producing noise. The severity field is the primary filter: LOW entries should be expected to generate low-signal-to-noise; analysis should weight MEDIUM and HIGH entries preferentially.

An alternative: require that entries include a `Suggested Fix` before they are classified as MEDIUM or HIGH. This forces Copilot to think through the improvement, not just complain.

## 5. Periodic Analysis — How Claude Synthesizes Friction

Claude (Cowork) is the natural analyzer. It already has access to the workspaces/ directory. A periodic review can be a scheduled task or a manual trigger.

### 5.1 Proposed Analysis Flow

```
Trigger: Monthly (scheduled) or manual ("synthesize kit friction")

1. Read all *-friction-log.md files from C:\Users\schur\workspaces\.kit-feedback\
2. Parse all FRICTION-ENTRY blocks
3. Deduplicate: group entries by Location + FrictionType (same rule, same type = same signal)
4. Score each unique signal: count × severity weight (LOW=1, MEDIUM=3, HIGH=9)
5. Rank signals by score (descending)
6. For the top N signals (suggested: top 5–8):
   a. Quote the original entry descriptions
   b. Assess whether the signal has already been addressed in the current kit version
   c. Draft an improvement brief in the format of KIT-IMPROVEMENT-BRIEFS.md
7. Write output to:
   C:\Users\schur\workspaces\.kit-feedback\SYNTHESIS-REPORTS\SYNTHESIS-<DATE>.md
8. Surface to Stephen with: "X new friction signals found; Y already addressed; top Z drafted as improvement briefs"
```

### 5.2 Severity Weighting Rationale

The 1/3/9 scoring (LOW/MEDIUM/HIGH) is borrowed from QFD (Quality Function Deployment) weighting — a common engineering practice for prioritizing competing requirements. A single HIGH entry outweighs three MEDIUM entries; a MEDIUM entry outweighs three LOWs. This prevents a large volume of LOW-severity noise from burying a single critical contradiction.

### 5.3 Deduplication Logic

Two entries are considered the same signal if:
- Same `Location` value (same file, same section) AND
- Same `Friction Type`

If the `Description` fields differ meaningfully (different aspects of the same rule are confusing), they should be treated as separate signals. Claude's judgment call on description similarity is acceptable here — this is analysis, not enforcement.

### 5.4 Already-Addressed Check

Before drafting an improvement brief, Claude checks the current kit version's file at the identified location to see if the issue was already resolved. This prevents the analysis from generating stale briefs for issues that have been fixed. The check should be: read the current content of the flagged file/section and assess whether the Suggested Fix from the friction entry is already implemented.

### 5.5 Output Format — Synthesis Report

The synthesis report in `SYNTHESIS-REPORTS/` should follow this structure:

```markdown
# Kit Friction Synthesis — YYYY-MM-DD

> Kit Version at time of analysis: vX.Y.Z
> Friction logs read: <list of files>
> Total entries analyzed: N
> Date range covered: YYYY-MM-DD to YYYY-MM-DD

## Summary

- Total unique signals: N
- Already addressed in current kit: N
- New signals requiring improvement briefs: N
- Signals below threshold (LOW-only, score < 3): N (listed in Appendix)

## Improvement Briefs — Ready for GPT

[one brief per signal, following KIT-IMPROVEMENT-BRIEFS.md format]

## Appendix — Low-Score Signals

[table of LOW-severity entries that did not meet the threshold for a brief]
```

## 7. Design Decisions and Tradeoffs

### 7.1 Why Append-Only Files (Not Timestamped Individual Files)

**Alternative considered:** One file per session, timestamped (e.g., `PIE-2026-04-16-001.md`).

**Decision: Append-only per repo.**

Append-only keeps the directory from accumulating hundreds of files. Analysis is simpler (read N files, not N×sessions files). The repo name in the filename is sufficient to correlate entries with their source. The `Date` field inside each entry provides temporal resolution.

### 7.2 Why Local Directory (Not a Git Repo)

**Alternative considered:** A dedicated `vibe-kit-feedback` GitHub repo that Copilot pushes to after each session.

**Decision: Local directory.**

A git repo for feedback would require Copilot to have push access, handle merge conflicts if two sessions run simultaneously (unlikely but possible), and add infrastructure complexity. The local directory is simpler, always available offline, and sufficient for a single-developer setup. If Stephen ever works across multiple machines, this design would need revisiting.

### 7.3 Why Not Use Consumer Repo Docs for Friction Logs

**Alternative considered:** A `KIT-FRICTION-LOG.md` inside each consumer repo's docs/vibe-coding/ directory, submitted with git commits.

**Decision: Shared external directory.**

Putting friction logs inside consumer repos creates three problems: (1) the kit subtree pull could overwrite a template; (2) friction logs would be committed to consumer repo history (noise); (3) cross-repo analysis would require reading multiple repos. The external shared directory avoids all three.

### 7.4 Copilot Instruction Risk — Over-Logging

If Copilot logs friction too liberally, the signal-to-noise ratio degrades. Mitigations:
- The severity field (LOW/MEDIUM/HIGH) lets analysis weight-filter noise
- Requiring a `Suggested Fix` for MEDIUM/HIGH entries acts as a quality gate
- The instruction explicitly says "write nothing" if no friction occurred (silence is a valid and expected output)

### 7.5 Privacy and Confidentiality

Friction entries will inevitably reference project-specific context (e.g., "during session PIE-AUDIT-001, the rule about..."). The `.kit-feedback/` directory is local and not committed to any repo. No confidential project information leaves the machine. If the directory is ever backed up to cloud storage, entries should be reviewed for project-specific content first.

## 9. Relationship to Existing Kit Improvement Workflow

This system complements — does not replace — the existing improvement workflow:

```
EXISTING: Stephen notices → creates improvement brief → GPT composes prompt → Copilot executes
NEW ADDITION: Copilot notices → logs to .kit-feedback → Claude synthesizes → improvement briefs → existing workflow continues
```

The output of the friction synthesis (improvement briefs in the Synthesis Report) feeds directly into the `KIT-IMPROVEMENT-BRIEFS.md` workflow already established in kit-workspace. The friction system is an **upstream signal generator** for the existing improvement pipeline.

- **Path:** `.kit-feedback/FRICTION-LOG.local.md`

- **Maximum:** One entry per session
- **Severity threshold:** MEDIUM or HIGH only (LOW excluded from pilot)

- **Explicit exclusions:** Normal coding retries, flaky tests, path typos, selector fixes, repo-specific discoveries, generic implementation churn
- **Synthesis model:** Manual or on-demand cross-repo review; no scheduled automation yet
